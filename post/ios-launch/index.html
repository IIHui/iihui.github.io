iOS 启动时长优化

创建: 2021-12-08   |   更新: 2099-12-08   |   分类: iOS     |   字数: 4787   |   阅读: 10 分钟

基本概念

  • Cold Launch: 一般翻译为冷启动。表示杀掉App，然后重启设备（重新启动几分钟后让它冷却），然后再打开App。
  • Warm Launch：杀掉app，然后再打开App。
  • Hot Launch：App从后台切到前台。

在这三种启动方式中，最重要的是冷启动。因为用户对你的应用程序的第一印象将是冷启动，而且也是最耗时的。启动时长的计算是通过两个关键指标来完成的。

  • Pre-main Time：执行应用程序的第一个函数（main函数）之前的时间。此时间保留给你的应用程序正确启动所需的系统绑定资源。
  • Post-main Time: 当您的应用程序从 UIApplicationMain（或主函数）控制到呈现给您的用户的第一个可交互屏幕的时间。简单来说，就是从 AppDelegate / Main 开始渲染第一帧所需的时间。

什么可能导致启动时间变慢？可能有多种原因，但常见的是：

  • Dynamic linking (dyld)
  • Business logic done before showing the first screen interact-able

动态链接的问题：将许多框架动态链接起来会使 iOS 系统进行许多绑定，从而增加 pre-main 时间。如果你使用 Cocoapods / Carthage，可能会有大量的依赖，然后甚至可能触发传递依赖（基础的依赖包括它们的嵌套的依赖等）。

这有可能会在启动时积累大量链接，然后影响启动时间。查看动态链接库数目的方法是在目标scheme的环境变量中添加 DYLD_PRINT_LIBRARIES，并将其值置为1。这样就会在控制台打印动态链接库的详细信息：

影响App冷启动时长的因素

  • 设备类型：设备越老，启动时长越长。
  • iOS 版本：iOS 13 做了一些优化，缓存了应用程序的动态依赖项，这显着改善了温启动，之前的 iOS 版本没有受益于这个改进。
  • 依赖的动态库数目：依赖的动态库越多，冷启动的时间越长。
  • 在使第一个屏幕可见之前要做的事情：如果正在加载外部依赖项（例如：分析框架）、初始化数据层（例如：CoreData），这将需要一些时间才能让您的用户看到第一个屏幕并可能影响启动。

Pre-main阶段的优化

  • 将第三方的动态库都改造成静态库。动态库可能是应用程序总启动时间中最长的部分之一。 Apple 建议最多只使用六个非系统framework，但对于包括我们在内的大多数现代应用程序来说，这是不现实的。可以看看是否能通过用静态版本替换动态版本或直接编译它们的源来删除正在使用的任何动态库。

  • 如果当前是使用Cocoapods管理库，可以使用名为 Amimono 的插件显着缩短我们的启动时间，该插件将符号从我们的 pod 复制到主应用程序可执行文件中。

  • 如果使用Carthage或手动管理动态库，您可以尝试Eric Horacek描述的这种策略：将单个库转换为静态库，然后通过单个动态框架将它们链接到应用程序中。我没有亲自尝试过，所以我不能保证这是个好主意。

  • 使用+initialize替代+load对Objective-C类进行初始化。参考自

  • 删除未使用掉的 Objective-C 代码。编译到应用程序中的每个类都会减慢变基/绑定和 Objective-C 初始化时间，因此请删除不再使用的类。

Post-main阶段的优化

  • Be Lazy：如果在启动期间正在做一些可能会在以后发生的事情，可以稍后再做。一个简单的例子是等待创建一个特定的视图控制器，用户实际点击一个按钮才需要显示它。在某些情况下，Swift 的lazy属性使这变得非常容易：

    private lazy var settingsViewController: MySettingsViewController = {
    return MySettingsViewController()
}()
  
private func onSettingsButtonPressed() {
    present(self.settingsViewController, animated: true)
}

  • 使用后台线程：如果一些事情不需要在主线程操作，可以将其放到后台线程，主线程继续做UI相关操作。但千万要注意线程安全问题。

  • 在行为上妥协。归根结底，您的应用程序在启动时尝试执行的操作越多，启动速度就越慢。与您的团队协商，看看您是否愿意更改或删除任何面向用户的内容，以缩短启动时间。也许对你的用户来说，拥有一个快速加载的简单初始 UI 更有价值，而不是一个充满平滑渐变、阴影和有用数据但需要永远显示的用户界面。
