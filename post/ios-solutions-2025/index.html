No Symbolic Information Found

在一台非打包的iMac上使用命令symbolicatecrash xxx1.crash xxx2.dSYM -o xxx1.log对崩溃日志进行符号化，执行完后出现了如下错误：

No Symbolic Information Found

最开始怀疑是dsym文件和崩溃日志的uuid不同造成的。但使用命令dwarfdump --uuid xxx.dSYM得到dsym文件的uuid和崩溃日志中的uuid是相同的，这就排除了是因uuid不一致而导致的上述报错。排查过程中发现，若使用atos并指定dsym目录下的target进行符号化时，得到的结果却是正常的：

atos -o ST/dsymDir/xxx1.dSYM/Contents/Resources/DWARF/targetxxx -l 0x1c6573000 0x00000001c6a0da80

在排查过程中又意外发现，使用symbolicatecrash时若按照atos命令那样指定dysm目录下的target也是可以符号化成功的，具体命令如下（其中xxx表示dsym文件名，xxx1表示崩溃日志文件名，xxx2表示target名）：

symbolicatecrash xxx1.crash xxx.dSYM/Contents/Resources/DWARF/xxx2  -o xxx1.log

此外，进一步排查又发现使用symbolicatecrash命令进行符号化时若添加-d选项（表示明确指定添加额外的dsym文件，匹配uuid时用到）也是能够正常符号化崩溃日志：

symbolicatecrash xxx1.crash -d xxx.dSYM -o xxx1.log

根据添加-d选项后终端的输出日志，发现若使用mdfind查找uuid对应的dsym文件时无任何输出，也就说找不到uuid对应的dsym文件：

mdfind "com_apple_xcode_dsym_uuids == A7C6DA83-B3C1-3E0C-9734-0C59CD5032C8"

这是因为Spotlight索引未包含dsym文件的元数据，可以通过如下方案解决。经过验证，第一次添加元数据标签成功后，后续再添加dsym文件时不再需要重复添加元数据标签（因为直接使用mfind可以查到到相应的dsym文件）：

# 替换为你的实际 UUID 和文件路径
UUID="A7C6DA83-B3C1-3E0C-9734-0C59CD5032C8"
DSYM_PATH="/path/to/xxx.dSYM" # 需要使用绝对路径

# 添加元数据标签
xattr -w com.apple.metadata:com_apple_xcode_dsym_uuids "$UUID" "$DSYM_PATH"

# 验证是否添加成功
mdls -name com_apple_xcode_dsym_uuids "$DSYM_PATH"

添加元数据后，再刷新索引即可：

# 刷新特定文件
mdimport "$DSYM_PATH"

# 或刷新整个目录
mdimport ~/Library/Developer/Xcode/Archives  # 假设dSYM在此目录

# 重建整个索引（耗时较长）
sudo mdutil -E /

Xcode 16 编译问题

升级至Xcode 16后编译原工程出现了如下错误：

#include <sys/_types/_u_char.h>'; 'u_char' must be declared before it is used

这是因为苹果在iOS 17/macOS 14 SDK中，移除了隐式类型声明，要求所有的类型如u_int32_t，u_char等需要从适当的系统模块导入。这也意味着现在需要显式地导入包含这些类型定义的头文件，即添加：

#include <sys/types.h>

解决了上面错误，又出现了第二个报错：

Declaration of 'sa_family_t' must be imported from module 'Darwin.POSIX.sys.types._sa_family_t' before it is required

同样地，也需要显式导入如下头文件：

#include <sys/socket.h>

升级到Xcode 16.4后，发现不能调试iOS 13真机，连上手机并手动信任后出现如下错误：

Could not locate device support files。You may be able to resolve the issue by installing the latest version of Xcode from the Mac App Store or developer.apple.com.

这是因为Xcode 16将支持的最低统版本限制到iOS 15了。而以前在使用低版本的Xcode时，当真机连上Xcode后，会自动检测本地是否已有相应系统的支持文件，若没有会自动下载对应目录：

/Applications/Xcode.app/Contents/Developer/Platforms/iPhoneOS.platform/DeviceSupport

因为Xcode 16不再支持iOS 15以下设备了，所以现在不会自动下载iOS 15以下的设备支持文件了。此时，需要手动下载低版本iOS系统的支持文件，当前的解决方案是到这里下载对应iOS版本的DeviceSupport文件，然后解压并移动到：

/Applications/Xcode.app/Contents/Developer/Platforms/iPhoneOS.platform/DeviceSupport

接着再重启Xcode，就可以正常调试低版本手机了。

覆盖安装Xcode

从苹果官网下载新版Xcode 16覆盖之前从App Store上安装的低版本Xcode时，出现如下报错：

The operation can’t be completed because some items had to be skipped. For each item, choose File > Get Info, make sure “Locked” is deselected, and then check the Sharing & Permissions section. When you are sure the items are unlocked and not designated as Read Only or No Access, try again.

可以通过如下方式解决：

# 解锁所有文件（移除用户不可更改标志）
sudo chflags -R nouchg /Applications/Xcode.app

# 将所有权赋予当前用户（替换YOUR_USERNAME为你的用户名）
sudo chown -R YOUR_USERNAME:staff /Applications/Xcode.app

# 授予用户完全读写权限
sudo chmod -R u+rw /Applications/Xcode.app

示例：如果用户名为john，则上面命令为sudo chown -R john:staff /Applications/Xcode.app。

UICollectionView

升级Xcode 16编译原工程后，出现如下报错：

Expected dequeued view to be returned to the collection view in preparation for display. When the collection view's data source is asked to provide a view for a given index path, ensure that a single view is dequeued and returned to the collection view. Avoid dequeuing views without a request from the collection view. For retrieving an existing view in the collection view, use -[UICollectionView cellForItemAtIndexPath:] or -[UICollectionView supplementaryViewForElementKind:atIndexPath:]

排查发现是因为在非cellForItemAt方法中使用了如下代码：

if let indexPath = indexPath, let cell = collectionView.dequeueReusableCell(withReuseIdentifier: xxxIdentifier, for: indexPath) as? TestCell {
   //
}

应该改成如下方式：

collectionView.cellForItem(at: index) as? TestCell

UINavigationController中查找特定View

extension UINavigationController {
    /// 获取导航控制器的 UIViewControllerWrapperView（仅用于调试）
    func debugFindWrapperView() -> UIView? {
        // 递归查找特定类名的视图
        func findView(in view: UIView, className: String) -> UIView? {
            if "\(type(of: view))" == className {
                return view
            }
            for subview in view.subviews {
                if let result = findView(in: subview, className: className) {
                    return result
                }
            }
            return nil
        }
        
        #if DEBUG
        return findView(in: self.view, className: "UIViewControllerWrapperView")
        #else
        return nil  // 正式环境永远返回 nil
        #endif
    }
}

fileprivate extension UINavigationController {
    /// 获取特性tag的View
    func debugFindWrapperView(with tag: Int) -> UIView? {
        // 递归查找特定类名的视图
        func findView(in view: UIView, tag: Int) -> UIView? {
            if view.tag == tag {
                return view
            }
            for subview in view.subviews {
                if let result = findView(in: subview, tag: tag) {
                    return result
                }
            }
            return nil
        }
        return findView(in: self.view, tag: tag)
    }
}

BIOCPROMISC: Operation not supported

为iOS设备创建虚拟接口：

rvictl -s xxx  # 其中xxx表示设备id

关闭虚拟接口：

rvictl -x xxx # 其中xxx表示设备id

将设备连接到mac上后可以使用如下命令查看设备id：

idevice_id -l

在macOS上使用rvictl -s为iOS设备创建虚拟接口(如 rvi0)后，Wireshark抓包时出现BIOCPROMISC: Operation not supported on socket错误，是因为Wireshark默认尝试启用混杂模式（Promiscuous Mode），但rvi0虚拟接口不支持此模式。以下是几个解决方案：

禁用混杂模式（推荐）

打开Wireshark，点击菜单栏的设置按钮，然后找到rvi0接口，取消勾选Promiscuous Mode即可，这是最简单的解决方案。

通过命令行工具捕获（避免 Wireshark GUI）

使用tshark（Wireshark的命令行工具）并显式禁用混杂模式：

sudo tshark -i rvi0 -p -w output.pcap

其中-p表示禁用混杂模式。-w output.pcap表示将捕获结果保存到文件，之后可在Wireshark中分析。

使用tcpdump捕获后导入Wireshark

sudo tcpdump -i rvi0 -s0 -w capture.pcap

按 Ctrl+C 停止捕获后，用Wireshark打开 capture.pcap 文件分析。

修改Wireshark全局设置

编辑Wireshark配置文件：

1、关闭Wireshark。

2、创建或修改配置文件：

nano ~/.config/wireshark/preferences

3、添加以下行：

capture.disable_promiscuous_mode: TRUE

4、保存后重启Wireshark。

原因说明

  • rvictl创建的rvi0接口本身已完整捕获 iOS 设备的流量，无需混杂模式。
  • Wireshark默认启用混杂模式，但macOS的BPF系统对虚拟接口支持有限，导致BIOCPROMISC错误。

通过上述任一方法禁用混杂模式即可解决该错误，正常捕获iOS设备流量。推荐优先使用方法1或方法 2。

iOS 18 unsatisfied (Local network prohibited)

使用Xcode 16编译App，使用某个上传功能时出现如下错误：

NSURLErrorNWPathKey=unsatisfied (Local network prohibited)

这个错误表明应用被系统阻止访问本地网络，这是因为iOS 18有一些新变化：

  • 进一步加强了网络隔离策略
  • 强化了本地网络访问的权限控制
  • 对mDNS/Bonjour服务发现更严格限制

解决方案是要在工程的info.plist添加如下内容：

<key>NSLocalNetworkUsageDescription</key>
<string>APP_NAME需要访问本地网络以发现附近的智能设备</string>

<key>NSBonjourServices</key>
<array>
    <string>_http._tcp</string>
    <string>_printer._tcp</string>
</array>

Task冲突

在某个Swift文件开头引入了PomiseKit，即import PromiseKit。然后，使用新并发模型中的Task{ }时，却出现如下编译错误：

'Task' cannot be constructed because it has no accessible initializers

这是因为在PromiseKit中也定义了一个Task，所以会优先使用PromiseKit里的这个Task，而不是Swift标准库中的Task。解决方案是使用标准并发模型的内部命名空间，即_Concurrency：

_Concurrency.Task {
   //
}

查看targert设置

升级了Xcode 16之后发现同一个工程的不同target有一个编译成功，另一个编译失败。根据现象，显然是因为不同target的编译设置不一样导致的，为了查看不同的target的编译设置，可以使用如下命令：

xcodebuild -workspace xxx.xcworkspace -scheme "xxxST" -configuration Debug -showBuildSettings > ~/Downloads/st_build_settings.txt

然后使用文本对比工具比对不同target的编译设置，根据结果找出不同的设置项。

编译时删除${BUILT_PRODUCTS_DIR}/xxx/xxx.framework

执行pod update后发现pbxproj中的"${BUILT_PRODUCTS_DIR}/xxx/xxx.framework"被删了，使用如下命令可以恢复：

# 1. 完全清除现有 Pods 集成
pod deintegrate

# 2. 清理缓存
pod cache clean --all

# 3. 重新安装
pod install

# Podfile

        # config.build_settings['WATCHOS_DEPLOYMENT_TARGET'] = '7.0'
      end
    end
  end
end

PBXFileSystemSynchronizedRootGroup

接手别人的项目后使用pod update命令更新依赖，出现如下错误：

RuntimeError - `PBXGroup` attempted to initialize an object with unknown ISA `PBXFileSystemSynchronizedRootGroup` from attributes: `{"isa"=>"PBXFileSystemSynchronizedRootGroup", "exceptions"=>["416D36782ECD6C14005419D5"], "path"=>"demo", "sourceTree"=>"<group>"}`

这是因为cocoapods版本太低（实际是cocoapods的Xcodeproj版本太低），无法识别在高版本Xcode上创建的项目。需要将在高版本Xcode上已创建工程的根目录由Folder转为Group（打开项目在根目录上右键会出现相应选项）。若按上面方法修改后执行pod update后还是出现如下报错:

[!] [!] Xcodeproj doesn't know about the following attributes {"minimizedProjectReferenceProxies"=>"1", "preferredProjectObjectVersion"=>"77"} for the 'PBXProject' isa.

这是因为低版本的Xcodeproj无法识别上面在高版本上支持的两个新属性，此时需要把文件project.pbxproj（在finder中右键xxx.xcodeproj—>显示包内容—>打开project.pbxproj）中的对应内容删除：

minimizedProjectReferenceProxies = 1;
preferredProjectObjectVersion = 77;

如果按上面方式删除上面两个字段，还是出现如下报错：

/Library/Ruby/Gems/2.6.0/gems/xcodeproj-1.21.0/lib/xcodeproj/project.rb:228:in `initialize_from_file': [Xcodeproj] Unknown object version (56). (RuntimeError)

则需要修改project.pbxproj文件中的objectVersion为支持的版本（查看历史支持的版本），才能编译正常。

发布和编译版本

使用xcodebuild命令进行打包时，可以指定编译版本和发布版本。具体命令如下，其中的变量targetName表示target，而bundleShortVersion表示发布版本，bundleVersion表示编译版本：

xcodebuild -quiet -workspace demo.xcworkspace -scheme "${targetName}" MARKETING_VERSION=${bundleShortVersion} CURRENT_PROJECT_VERSION=${bundleVersion}

若需要在某个目录下删除非ipa和非dSYM的其它所有文件，可以使用如下命令：

# 删除非.ipa和非.dSYM文件
find "$build_path" -type f ! -name "*.ipa" ! -name "*.dSYM" -depth 1 -delete

某个变量取两个环境变量的值，下面的变量是若$2为空，则取$demo的值：

bundleShortVersion="${2:-$demo}"

cannot be found -Xlinker

修改工程Build Settings中的Other Linker Flags值，去除了-force_load紧接着的下一行，接着重新编译时出现如下错误：

Build input file cannot be found: '/xxx/-Xlinker'. Did you forget to declare this file as an output of a script phase or custom build rule which produces it?

这是因为-all_load或-force_load标志的作用都是加载静态库中所有文件，但-all_load作用于所有的库，而-force_load后面必须要指定具体的文件，现在删除了其后的文件，编译就出现了错误。

CodeSign Error

iPad TabBar显示

在iOS 18的iPad上，用户反馈TabBar到了顶部，而不像手机那样显示在底部。可以通过修改UITabBarController中的traitOverrides属性为compact来解决：

// 兼容iOS 18以上系统，将tabbar显示在底部
if UIDevice.current.userInterfaceIdiom == .pad {
    if #available(iOS 18.0, *) {
        traitOverrides.horizontalSizeClass = .compact
    }
}

libswiftUniformTypeIdentifiers

在Xcode 16上调试iOS 13时出现启动闪退，显示的错误信息如下：

dyld: Library not loaded: /usr/lib/swift/libswiftUniformTypeIdentifiers.dylib
  Referenced from: /private/var/containers/Bundle/Application/4C36D45A-4944-41F6-8184-29C495C593AE/xxx.app/xxx
  Reason: image not found
dyld: launch, loading dependent libraries
DYLD_LIBRARY_PATH=/usr/lib/system/introspection
DYLD_INSERT_LIBRARIES=/Developer/usr/lib/libBacktraceRecording.dylib:/Developer/usr/lib/libMainThreadChecker.dylib:/Developer/Library/PrivateFrameworks/DTDDISupport.framework/libViewDebuggerSupport.dylib:/usr/lib/libMTLCapture.dylib

出现上面错误是因为Xcode 16编译应用时链接了libswiftUniformTypeIdentifiers.dylib，该库是Swift对 UniformTypeIdentifiers框架的overlay，而这个框架仅在iOS 14+系统里存在。在iOS 13设备上运行，dyld找不到该库，所以出现直接崩溃。即使没有显式import UniformTypeIdentifiers，只要代码中（包括依赖库）使用了UTType等API，编译器就会自动链接这个库。

第一种解决方案是条件化所有UniformTypeIdentifiers相关代码，即用#available把相关逻辑包裹起来，不要在文件顶部无条件import UniformTypeIdentifiers。

// 正确做法：需要时才导入，并用available保护
if #available(iOS 14, *) {
    import UniformTypeIdentifiers
    // 使用 UTType 等
}

第二种解决方案是弱链接UniformTypeIdentifiers：

  • 打开Xcode项目，选择target。
  • 进入General → Frameworks, Libraries, and Embedded Content。
  • 点击 +，搜索并添加 UniformTypeIdentifiers。
  • 将该framework的Embed方式设为Do Not Embed。
  • 把它右侧的链接选项改为Optional（而不是Required）。

即使做了上面操作，libswiftUniformTypeIdentifiers.dylib仍可能被标记为必需。需要手动在链接器参数中将它改为弱链接，进入Build Settings → 搜索Other Linker Flags，添加下面一行：

-weak-lswiftUniformTypeIdentifiers
# 或者
-Wl,-weak-lswiftUniformTypeIdentifiers

最后，使用如下命令检查ipa中确认libswiftUniformTypeIdentifiers.dylib旁边显示的是weak而不是常规引用：

otool -L YourApp.app/YourApp | grep UniformTypeIdentifiers

URLSessionConfiguration

使用URLSession时，其配置URLSessionConfiguration有default和ephemeral两个选项，具体区别如下：

特性 default ephemeral
缓存存储 磁盘 + 内存 仅内存
Cookie 存储 持久化到磁盘 仅内存，会话结束即清除
认证凭证 持久化到钥匙串（磁盘） 仅内存，会话结束即清除
跨 App 重启保留数据 是 否
内存占用 相对较低（部分存在磁盘） 相对较高（全在内存）
典型用途 常规请求，需要缓存/登录态 隐私模式，临时会话

invalidateAndCancel和finishTasksAndInvalidate

若要终止URLSession任务，有invalidateAndCancel和finishTasksAndInvalidate两个方法，核心区别在于对当前正在执行的任务的处理方式，以及会话失效的时机，一个是「立即强制终止」，一个是「优雅地等待完成」。

当invalidateAndCancel调用后，会话会立即标记为无效，所有未完成的任务（包括正在网络传输的和队列中等待的）都会被强制取消。被取消的任务会收到 URLError.cancelled错误，其completionHandler会立即被执行（或delegate方法会被触发）。一旦调用，该会话不能再创建新的请求，否则会出现崩溃或报错。

当finishTasksAndInvalidate被调用后，会话会立即停止接收新的任务（之后调用dataTask会失败），但允许当前已经在传输中的任务继续执行，直到它们自然完成（成功或失败）。只有当所有现有的任务全部执行完（调用完各自的completionHandler）后，会话才会正式释放资源。如果队列中有尚未开始的任务（resume()还没调用或处于等待状态），这些任务会被取消，但已经开始传输的任务会保持运行。

特性 invalidateAndCancel finishTasksAndInvalidate
正在传输的任务 立即取消（中断连接） 继续执行，直到完成
队列中等待的任务 立即取消 立即取消（不会启动）
会话释放时机 立即释放资源 等待所有活动任务完成后才释放
任务回调错误 任务收到cancelled错误 正常收到success或服务器返回的错误
内存泄漏风险 较低（立即释放） 较高（如果任务长时间挂起，会话不会被释放）
  • 内存泄漏警告：如果调用了finishTasksAndInvalidate()，但某个下载任务因为网络问题一直在重试或卡死，那么Session永远不会释放，从而导致内存泄漏。因此，通常建议在应用进入后台时使用finishTasksAndInvalidate，同时设置合理的超时时间（timeoutIntervalForRequest）。
  • 调用后禁止复用：无论调用哪个方法，该URLSession实例在失效后都不能再被复用。如果还需要发起请求，必须重新创建URLSession实例。

zlib-ng.h file not found

依赖SSZipArchive时，出现编译错误：

mz_strm_zlib.c:16:12: error: 'zlib-ng.h' file not found

是因为SSZipArchive依赖的minizip-ng库需要zlib-ng，编译时找不到zlib-ng.h头文件。最后发现是pod_target_xcconfig设置有问题，其完整设置如下：

 s.pod_target_xcconfig = { 'DEFINES_MODULE' => 'YES',
    'GCC_PREPROCESSOR_DEFINITIONS' => 'HAVE_ARC4RANDOM_BUF HAVE_ICONV HAVE_INTTYPES_H HAVE_PKCRYPT HAVE_STDINT_H HAVE_WZAES HAVE_ZLIB ZLIB_COMPAT' }

上面设置中的DEFINES_MODULE' => 'YES'是告诉Xcode将这个Pod构建为模块（module），并生成 .modulemap文件，即：

  • 让Swift能通过import YourPod引入这个库（即使是纯C/ObjC库也能生成module map）。
  • 开启CLANG_ENABLE_MODULES，允许库内用@import 语法。
  • 如果这个库原本不支持module，加上这行能避免Swift编译时出现无法导入的错误。

而'GCC_PREPROCESSOR_DEFINITIONS'后面的一串内容，它相当于在所有源文件中执行了：

#define HAVE_ARC4RANDOM_BUF 1
#define HAVE_ICONV 1
#define HAVE_INTTYPES_H 1
#define HAVE_PKCRYPT 1
#define HAVE_STDINT_H 1
#define HAVE_WZAES 1
#define HAVE_ZLIB 1
#define ZLIB_COMPAT 1

这些宏通常用于移植或编译开源C库时（尤其是OpenSSL、libz、libiconv等）做特性检测。每一个宏的定义解释如下：

宏 含义
HAVE_ARC4RANDOM_BUF 平台提供arc4random_buf函数（iOS/macOS都有），用于生成加密安全的随机数
HAVE_ICONV 支持iconv字符编码转换（系统自带libiconv）
HAVE_INTTYPES_H 存在<inttypes.h>头文件，定义跨平台的固定宽度整数格式宏（C99 标准）
HAVE_PKCRYPT 支持PK加密相关功能（多见于OpenSSL/libressl等库的兼容检测）
HAVE_STDINT_H 存在<stdint.h>头文件（C99），提供int32_t、uint64_t等类型
HAVE_WZAES 可能表示使用WZ（某定制版本？）的AES实现或特定补丁
HAVE_ZLIB 链接zlib，支持数据压缩
ZLIB_COMPAT 表示zlib接口与原版zlib兼容（常见于使用zlib-ng或兼容层时）

上面的宏定义右侧没有等号，在Xcode中解析出来的默认值是1，若想自定义为其它值可以进行如下设置：

'GCC_PREPROCESSOR_DEFINITIONS' => 'HAVE_ARC4RANDOM_BUF=1 HAVE_ICONV=2 CUSTOM_VALUE=hello'

为了避免字符串拼接出现错误，在CocoaPods中更推荐用数组：

s.pod_target_xcconfig = {
  'DEFINES_MODULE' => 'YES',
  'GCC_PREPROCESSOR_DEFINITIONS' => [
    'HAVE_ARC4RANDOM_BUF=1',
    'HAVE_ICONV=2',
    'CUSTOM_BUFFER_SIZE=4096',
    'LIB_VERSION="1.2.3"'  # 注意：带引号的字符串值需要转义
  ]
}

如果要定义字符串值的宏，需要小心引号的转义问题：

# 可能出问题（取决于Ruby字符串处理方式）
"VERSION_STRING=\"1.0\""    # 双引号里只转义一次，到 xcconfig就变成 VERSION_STRING=1.0

# 正确
'VERSION_STRING="1.0"'      # 单引号原样保留
# 或
"VERSION_STRING=\\\"1.0\\\"" # 双引号里双重转义

帧框加载已失败

今天在调试WKWebView时报错「帧框加载已失败」，同时看到的错误码为WebKitErrorDomain code 102。出问题的场景是在WKWebView中使用WKURLSchemeTask拦截网络请求，返回本地文件后遇到的错误。

最后发现是通过WKURLSchemeTask返回数据时，WebKit需要正确的HTTP响应头（Headers）来解析内容。如果信息不全，WebKit会认为响应无效并中断加载。首先不要使用URLResponse，因为它默认状态码为0，WebKit无法识别。必须创建HTTPURLResponse并明确设置200状态码，同时需要提供Content-Type和Content-Length这两个关键的Header，而此次出现错误是因为文件是html，但设置的Content-Type是错误的。

如果通过WKURLSchemeTask拦截的是一个子资源请求（例如，主页面是https://example.com，但iframe或图片的请求被自定义scheme如myapp://拦截并返回了本地数据），会构成了跨域请求。此时，需要在返回的 HTTPURLResponse的 headerFields 中，明确添加允许跨域的头部：

var headerFields = [String: String]()
// ... 设置 Content-Type, Content-Length ...
headerFields["Access-Control-Allow-Origin"] = "*" 
// 或者设置为你的主页面域名，如 "https://example.com"

此外，WKURLSchemeTask的生命周期需要精确管理。在WebView已经停止任务（调用stopURLSchemeTask）后，再调用didReceive或didFinish会导致崩溃。此外，如果在调用didReceive(URLResponse) 之前就发送了数据（Data），也会引发异常。解决方案如下：

  • 确保调用顺序正确：必须严格按照didReceive(response)->didReceive(data) -> didFinish()的顺序调用。
  • 处理任务停止情况：实现webView(_:stop:) 方法。当任务被停止时，应取消加载操作，并不再对该任务调用任何WKURLSchemeTask的方法。同时，确保所有对WKURLSchemeTask的调用都在主线程执行，以避免竞态条件。
